### 次要發現（診斷備忘）

1. **Next.js 路徑重疊但應該不是根因**：`app/api/clients/todos/route.ts` 跟 `app/api/clients/[id]/todos/route.ts` 內容**完全相同**（GET/POST）。Next.js static segment (`todos`) 優先於 dynamic (`[id]`)，所以：
   - `GET /api/clients/<uuid>/todos` → dynamic 版
   - `GET /api/clients/todos` → static 版（如果有人這樣打）
   - `PATCH /api/clients/todos/<uuid>` → `[todoId]/route.ts`（綠色 PATCH 實際打這裡）
   路徑衝突不影響 PATCH 分派。
2. **兩條路徑共用相同 Notion DB**（env `NOTION_TODO_DB_ID`）、相同 page IDs、相同 `notion.pages.update` 寫入點。
3. **UI render 都在同一個檔案** `app/marketing/page.tsx`（不是獨立 modal component）。這推翻了第十三場「綠色路徑是獨立 component / API」的假設。

### Archive / Delete 查證（針對使用者新觀察）

使用者反映：打勾後 Notion 那筆**被 archive / 刪除**、不是 checkbox 變 true。

全 repo grep 結果（`app/` 底下）：
- `rg "archived" app/` → **0 matches**
- `rg "pages\.delete|pages\.archive|\.archive\(" app/` → **0 matches**
- `rg "method: 'DELETE'"` in `app/marketing/page.tsx` → **0 matches**（整個前端沒送 DELETE）
- `rg "export async function DELETE" app/` → **唯一一處** `app/api/conversations/[id]/route.ts:50`（對話記錄 delete，跟待辦無關）
- `app/api/clients/todos/[todoId]/route.ts` PATCH handler 的 `updateProperties` 組合**只碰 `待辦/優先度/Status` 三個欄位**，payload 不含 `archived` 欄位

→ **程式碼端沒有任何地方呼叫 `notion.pages.delete`、`notion.pages.update({archived:true})` 或對待辦 DB 送 DELETE 請求**。如果使用者觀察到「被 archive」，根因必須來自：
  - (a) Notion 端 database automation（某個 rule 把「待辦=true」的 row 自動 archive）——這是**外部 / 非 code bug**
  - (b) `notion.pages.update` 回傳的 response 顯示 `archived: true` 但 SDK 沒送那個 flag——理論上不會發生，但 Step 2 新增的 Notion response log 會驗證
  - (c) 使用者看到的 row「消失」其實是 DB view 的 filter（例如「待辦 is not checked」）把打勾完的 row 藏起來——視覺上像 archive 但其實 row 還在

## 根因假設

### ✅ 主選：**A — 兩條路徑共用 notion 寫入點**

**證據**：PATCH URL、handler、notion call site **完全一樣**。只有 UI handler 名稱不同（`handleAnimatedToggleTodo` vs `handleToggleTodo`）和 payload body（永遠 `true` vs `toggle`）。

**推翻 B**：綠色沒有獨立的 notion 寫入點。
**推翻 C**：綠色 `handleToggleTodo` 明確呼叫 `fetch(..., PATCH)`，不是純 local state。

### 新增：**D — handler 動作錯位（archive/delete 而不是 update checkbox）**

**來源**：第四輪更新，使用者實測觀察「打勾後 Notion 那筆是被 archive / 刪除、不是 checkbox 變 true」。

**Code 端反證**：如上「Archive / Delete 查證」段。repo 完全沒有任何 archive / delete 呼叫路徑。
- 前端 `handleToggleTodo` 明確用 `method: 'PATCH'`，不是 DELETE
- 後端 PATCH handler 組的 `updateProperties` 裡完全沒有 `archived` key
- Notion SDK 的 `pages.update({ page_id, properties })` 不帶 `archived` 欄位時**不會** archive page

**推測實際機制**：
- D1（可能性最高）：Notion database 端有 **automation / rule**「`待辦` checkbox = true → archive this page」。Step 2 API 端新加的 `archived` / `in_trash` / `待辦checkbox` response log 可以驗證——如果 `notion.pages.update` 回來的 response 顯示 `archived: true`，而 SDK call 沒送那個 flag，就確定是 Notion 端 automation。
- D2：Notion 該 DB 有一個 view filter「待辦 is checked」被隱藏，打勾完後 row 被 filter 濾掉看起來像消失，但 row 其實還在、`archived: false`。請使用者在 Notion 換一個「全部顯示」的 view 確認。
- D3（code bug 可能性）：有第三條路徑在背景把 row archive。Grep 已排除，可能性極低。

### 分歧可能在以下更早的層（Step 2 instrumentation 要釐清的）

1. **綠色 fetch 有沒有真的發出 / 收到 2xx**？若 PATCH 500 或 throw，line 604 `throw` 會進 catch，`setClientTodos` 不跑，UI 不會變綠。但使用者說「前端文字不消失」——這暗示 `setClientTodos` 也沒跑，**可能是 fetch 根本沒收到 ok response，進了 catch**。這跟「Notion 那筆被 archive」同時成立的話，代表 PATCH 有打到後端、Notion 真的被改了（archive），但 API 因某種原因回了非 2xx（或 UI throw catch），UI 沒跑 optimistic update。
2. **payload body 差異**：若 `currentFlag` 前端 state 錯亂為 `true`，會送 `{ todoFlag: false }`。Step 2 instrumentation 會看到 UI 端實際送出 body 和 API 端接收到 body，能直接比對。
3. **`todoId` 值**：理論上是同一批 Notion page ID，但若綠色 `clientTodos` 拿到的 page.id 帶了前綴 / 被 trim / 編碼差異，API 端 `notion.pages.update({ page_id: params.todoId })` 可能拿錯 page 或失敗。Step 2 已 log `todoId`。
4. **Notion integration permissions**：若 integration 對「待辦事項」DB 只讀不寫，`notion.pages.update` 會 throw → catch → 500。Step 2 的 API `notion.pages.update threw` log 會捕捉到。

## 第五輪前置（使用者回報後要做的判斷）

根因判斷流程（**搭配第四輪新增的 `archived` / `in_trash` / `fullPayload` log 看**）：

| 觀察 | 根因判斷 |
|---|---|
| Notion 結果 = A (archive) + API log `archived: true` in returned | → Notion 端 **database automation** 規則把 `待辦=true` 的 row archive（**非 code bug**，是 Notion workspace 設定） |
| Notion 結果 = A (archive) + API log `archived: false` + 前端看不到 row | → Notion view **filter** 把打勾完 row 藏起來（視覺 bug，實際 row 還在、狀態正確） |
| Notion 結果 = B (checkbox=true) | → 後端寫入正確、前端 UI state 沒同步（`setClientTodos` 沒跑 / fetch 進 catch） |
| Notion 結果 = C (沒變) + API log 有 `notion.pages.update threw` | → Notion API 權限問題或 payload shape 錯 |
| Notion 結果 = C + API log 完全沒出現 | → PATCH 沒打到這個 route，可能打到其他（Network tab 的 URL 會揭露） |
| Console `[green-todo] PATCH request` body 含 `{todoFlag: false}` 但使用者說是打勾 | → `currentFlag` 前端 state 錯亂 |
| Console 完全沒有 `[green-todo]` log | → UI 事件沒觸發 / instrumentation 沒部署 |
| Network PATCH payload 含 `archived: true` 或 method 是 DELETE | → 推翻「repo 沒 delete path」前提，要重查（但 Grep 已確認不可能） |
