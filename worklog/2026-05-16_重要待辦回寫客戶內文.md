# 2026-05-16 重要事項/待辦事項 → 回寫客戶 Notion 內文 + 完成劃線

## 【背景】

行銷頁的「重要事項 / 待辦事項」（dashboard 頂部 + 客戶詳情頁最新進度）目前只用 Notion relation 把項目關聯到客戶，項目本身不出現在客戶 Notion 頁面內文。user 要：

1. 項目綁定客戶時、同步回寫一行到「該客戶 Notion 頁面內文（body）」
2. 項目完成後、內文那行加刪除線（只劃線、留著當記錄、不移除）
3. 範圍：重要 + 待辦 × dashboard + 詳情頁，兩處都比照

user 拍板（紀律 #4 四段 review 後）：
- 完成行為 = 只劃刪除線、留著
- 追蹤哪一行 = 存 block id（比照已驗證的開發 DB `行事曆ID` 模式、最穩）
- 收尾 = A（本回寫）獨立先 push，B（之前進度滑軌）、C（洽談卡重構）分開做

## 【做了什麼】

7 檔（3 新 4 改），**前端 page.tsx 零改**（全 server-side、最低風險）：

### 新增

- `scripts/_add-body-blockid-field.js`：在 IMPORTANT_DB + TODO_DB 各加 `內文BlockID` (rich_text)。dry-run + `--apply` 兩段式（鏡像 `_add-visit-fields.js`）。**已執行 --apply**：重要 DB 4→5 欄、待辦 DB 7→8 欄
- `lib/notion-body-log.ts`：共用 helper、全 best-effort（失敗只 console.error 不擋主流程，沿用 quick-log 慣例）
  - `formatBodyLine(kind, title)` → `[M/D ⭐ 重要] xxx` / `[M/D ☑ 待辦] xxx`（對齊 quick-log `[M/D 📞 洽談]`）
  - `appendClientBodyLine(clientPageId, text)` → `blocks.children.append` paragraph、回 block id
  - `saveBodyBlockId(itemPageId, blockId)` → 寫回項目頁 `內文BlockID`
  - `readBodyBlockId` / `setBodyBlockStrike(blockId, on)`（retrieve→改 annotations.strikethrough→update、保留文字）/ `strikeItemBodyLine`

### 改

- `app/api/important-items/route.ts` POST：建立後若 clientId → append 客戶內文 + 存 block id（重要、兩 surface 共用此路由）
- `app/api/todo-items/route.ts` POST：同上（dashboard 待辦、clientId from body）
- `app/api/clients/[id]/todos/route.ts` POST：同上（詳情頁待辦、clientId = params.id）
- `app/api/important-items/[id]/route.ts` PATCH：`status==='done'` → `strikeItemBodyLine(id, true)`
- `app/api/clients/todos/[todoId]/route.ts` PATCH：`todoFlag !== undefined` → strike(true)/unstrike(false)（待辦有復原 UX、雙向）

### 路由模型釐清（探出來的）

- 待辦全寫同一 `NOTION_TODO_DB_ID`；dashboard 建→`/api/todo-items`、詳情頁建→`/api/clients/[id]/todos`、完成都→`PATCH /api/clients/todos/[todoId]`（設 `待辦` checkbox）
- 重要寫 `NOTION_IMPORTANT_DB_ID`；兩處建→`/api/important-items` POST、完成→`PATCH /api/important-items/[id]`（設 `優先级`=DONE）
- `app/api/clients/todos/route.ts`（無 [id]）是死碼、無引用、未碰

## 【為什麼這樣做】

- **server-side、前端零改**：append/存id/劃線全塞進既有 POST/PATCH 路由，page.tsx handler 不動 → 風險最小、不碰已綠的前端
- **存 block id 不靠文字比對**：user 拍板。文字比對在「同名項目多筆 / 手動改過那行」會劃錯；block id 精準。比照開發 DB `行事曆ID` 既有模式
- **best-effort 不擋主流程**：沿用 quick-log / 帶看寫回慣例，Notion 失敗不讓建立/完成失敗
- **A 獨立先 push**：A 已驗證、獨立、可上線；B/C 未建（C 含 PG migration 大改）。綁一包未預覽 push = 重演今早 build fail 雷型，user 拍板 A 先走
