# 2026-05-16 帶看記錄卡（未收合）案名旁加列印鈕

## 【背景】

user 要在帶看記錄卡（未收合 = liked/預設展開態）案名旁加一個列印鈕，連結跟「新增帶看 modal 裡的列印」一樣。

data-first 探明：新增帶看的列印 = 開 i智慧 成交行情列印頁
`https://is.ycut.com.tw/case/report/market/redirect?caseIdx=<ycutCaseIdx>&empNo=<empNo>&autoPrint=1`，
關鍵參數 `ycutCaseIdx` = i智慧 **內部** caseIdx（跟對外 case-info URL 不同、由 userscript 查到、modal state `ycutCaseIdx`）。

**worst-case 揭露（非廉價安撫）**：查證 `Viewing` 型別 + `handleCreateViewing` + viewings 表 → **已存帶看根本沒存 ycutCaseIdx**（建立時沒送、表沒欄）。社區連結是對外網址、跟內部 caseIdx 不同（code 註解明寫）不能湊。→ 不是 5 分鐘加按鈕，是「真功能 + PG migration + 只往前」。user 拍板 A：真做、只往前生效。

## 【做了什麼】

PG migration + 5 檔（加性、只往前）：

- `scripts/_add-viewing-ycut-idx.js`：viewings 加 `ycut_case_idx TEXT NULL`（IF NOT EXISTS、冪等、鏡像 `_add-conversation-columns.js`）。**已執行**、26 列、加性無破壞
- `lib/types.ts`：`Viewing` 加 `ycut_case_idx?: string | null`
- `app/api/viewings/route.ts` POST：body 收 `ycutCaseIdx`、INSERT 加第 11 欄 `ycut_case_idx`（空字串 → null）
- `app/marketing/page.tsx` `handleCreateViewing`：POST body 帶 `ycutCaseIdx: ycutCaseIdx || undefined`（modal 既有 state）
- `app/api/clients/[id]/viewings` GET：SELECT 補 `ycut_case_idx`
- `components/ViewingCard.tsx`：import `Printer`；**未收合（預設/liked）態**案名旁，`viewing.ycut_case_idx` 有值才顯示「🖨 列印」鈕，onClick 開與 modal 完全相同的 i智慧 列印 URL（`empNo` 同樣 `NEXT_PUBLIC_YCUT_EMP_NO || 'C30419'`）。收合（disliked）態不放（user 指定「未收合」）

## 【為什麼這樣做】

- **複用 modal 完全相同 URL/empNo**：user 要「跟新增帶看列印一樣」，照抄那段邏輯零差異
- **只往前生效**：舊 26 筆無 caseIdx → 卡片不顯示列印鈕（同洽談/委託只往前模式）；不回填舊資料
- **只動未收合態**：user 明確「未收合」；收合灰列保持精簡
- **best-effort 無侵入**：ycut_case_idx 純附加欄、空值 null，不影響既有建立帶看流程

## 【驗證】

`npm run build` 綠。Smoke **不走 `/api/viewings` POST**（它會建真 Google Calendar 事件、污染行事曆）→ 改直接 PG 插一筆帶 `ycut_case_idx` 測試帶看、驗 GET 回傳該欄、清乾淨：
- 插入 → GET `/api/clients/[id]/viewings` 找到該列、`ycut_case_idx=SMOKEIDX123`、key 存在 ✅
- DELETE 清乾淨、0 殘留 ✅
POST 寫入路徑：build + 程式碼正確性（加 $11 param、鏡像既有 INSERT）保證；user 實際新增一筆帶看即可親驗。user 預覽通過「ok了」。

## 【下次接手要注意】

- **舊 26 筆帶看無列印鈕**：只往前生效、user 已知。若要回填需另外想辦法補 caseIdx（i智慧 lookup by 社區名、heavy、未做）
- **列印鈕只在未收合態**：disliked 收合列刻意不放
