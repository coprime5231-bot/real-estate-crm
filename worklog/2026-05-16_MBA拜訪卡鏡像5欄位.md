# 2026-05-16 MBA 拜訪卡鏡像 CRM 5 欄位（現況/坪數/開價/格局/社區）

## 【背景】

接續同日「CRM 拜訪卡屋主後加 5 欄位」（commit f634af2 + 修 build 99839ea）。user 要把同樣 5 個物件資訊（現況/坪數/開價/格局/社區連結）也顯示在 **MBA 現場手機拜訪卡**（`app/m/TodayTaskList.tsx`），出門到現場時也一眼看到。

關鍵架構差異（開工前已查證、跟 user 對齊）：

- **CRM 拜訪卡**：直接讀 Notion 開發 DB（`/api/dev`）→ 5 欄本來就在資料裡
- **MBA 拜訪卡**：讀今天 Google Calendar 行程（`/api/m/calendar/today` → `parseTodayEvents`）→ event 只有 summary/location/description，**沒有** Notion 物件欄位
- 但 CRM 排程建 event 時（`/api/dev/schedule-visit`）description 有埋一行 `Notion：https://www.notion.so/{32hex頁面id}` → MBA 可用此 id 反查 Notion 補資料

user 拍板：鏡像 5 欄 + 做法選「讀取時從 Notion 補」（沿用現有「帶看卡從 PG 撈 viewingExtras」的 enrichment 模式、非新發明）。

## 【做了什麼】

4 檔：

### 1. `lib/mba/visit-extras.ts`（新檔，鏡像 `viewings.ts`）

- `interface VisitExtras { occupancy, area, price, layout, web }`
- `getVisitExtrasFromDescription(description)`：
  - regex `notion\.so\/([0-9a-fA-F]{32})` 從 description 解 Notion 頁面 id（schedule-visit 寫的是無 dash 32 hex）
  - 解不出 → 回 null（舊事件 / 非 CRM 建、graceful degrade）
  - `notion.pages.retrieve({ page_id })` → 用 `extractText` / `extractSelectValue`（與 `/api/dev` mapPage 同一套）抽 `現況`(select)→occupancy、`坪數`→area、`開價`→price、`格局`→layout、`網頁`(url)→web
  - 5 欄全空 → 回 null（render 端不必判 5 個 falsy、也不顯示空殼標籤）

### 2. `lib/mba/calendar.ts`

- `TodayTask` interface 加 `visitExtras: VisitExtras | null` + import type

### 3. `app/api/m/calendar/today/route.ts`

- enrich map 內、`kind === 'visit' || 'visit_revisit'` 時呼 `getVisitExtrasFromDescription(c.description)`、`.catch(()=>null)`、掛成 `visitExtras`
- 與既有 `viewingExtras`（kind==='viewing'）完全平行的 Promise.all 模式

### 4. `app/m/TodayTaskList.tsx`

- 拜訪卡 body（location 之後、action 按鈕之前）插 visitExtras 區塊、guard `!done && (visit||visit_revisit) && t.visitExtras`
- 視覺鏡像既有 viewing extras 語言（MBA 手機大字大按鈕）：
  - 現況高亮：空屋=綠 `rgba(52,211,153,.18)`/`#6EE7B7`、自住=橘 `rgba(251,146,60,.18)`/`#FDBA74`、其他=灰
  - 坪數/開價/格局：灰標籤 `#3A4055`/`#C8CCD8`，坪數/開價 `endsWith` 防呆補「坪」「萬」
  - 社區：黃色可點 pill（`#FFD86B`、padding 12/20 minHeight 44、對齊 viewing 的「樂居」按鈕觸控規格）

## 【為什麼這樣做】

- **沿用 viewings enrichment 模式不另立**：MBA 已有「kind 判斷 → 補資料 → 掛 extras → render 端 guard」成熟 pattern，visit 完全平行接上、認知負擔最低、不引新架構
- **read-time 從 Notion 補（非排程時寫 event）**：選此法因 (a) 已排的舊 event 也能立刻有資料、零 migration (b) 不必定 event description 的 parse 格式（脆）。代價是每張拜訪卡 +1 Notion `pages.retrieve`、一天幾筆可接受、route 本來就 `force-dynamic`
- **欄位名不腦補**：開工前用 live Notion DB schema + 真實資料抽樣確認 `現況/坪數/開價/格局/網頁` 名稱型別（同 CRM 那輪），`getVisitExtrasFromDescription` 抽取邏輯與 production `/api/dev` mapPage 完全一致
- **全空回 null**：避免現場看到 5 個空殼標籤、符合北極星「不堆雜訊」

## 【下次接手要注意】

- **MBA 拜訪卡看不到標籤 ≠ bug**：先查該物件 Notion 開發 DB 那 5 欄是否有值、再查 event description 有沒有 `Notion：...{32hex}` 那行（舊 event / 非 CRM 排程建的沒有 → 永遠 null）
- **CRM/MBA 兩套抽取要同步**：`getVisitExtrasFromDescription`（MBA）與 `/api/dev` mapPage（CRM）抽同 5 個 Notion 欄位。Notion 欄位改名 / 加欄要兩邊一起改、否則 CRM 有、MBA 沒有
- **每張拜訪卡 +1 Notion retrieve**：若某天拜訪量暴增（幾十筆）感覺慢、可考慮 batch / 快取；目前一天幾筆不處理
- **現況高亮只認空屋/自住**：第三個 option 會灰底（同 CRM 端、user 已知拍板先不做）
- 純讀取：沒動 schema、沒動 schedule-visit、沒動 PATCH、沒動 PG
