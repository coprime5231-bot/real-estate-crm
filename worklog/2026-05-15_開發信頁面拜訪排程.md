# 2026-05-15 開發信頁面拜訪排程

## 【背景】

user 要解的問題：「開發」DB 內狀態 = `募集` 的客戶要大量實地拜訪、現有 CRM 只能印開發信標籤、沒有把拜訪流程接上來。出門前要在電腦上看清單 + 在 Google Calendar 排時間、目前是用 n8n 拼一段「待辦狀態變更 → 寫 Calendar」、想取代掉。

user 同時釐清了一個重要的職責分割：

- **CRM 拜訪卡** = 電腦前的「排程介面」（看哪些客戶要拜訪 + 建行事曆）
- **MBA 拜訪卡** = 現場手機的「操作介面」（人不在/找不到人/找到人了 三按鈕 → 改 Notion 狀態）

> 開發信跟拜訪是二條平行線各自運作、visitTodo 預設值由 MBA 新增募集物件時設定、CRM 端不互相觸發。

## 【做了什麼】

### Ticket #1 — Notion DB 加兩欄

- 用 Notion API 在「開發」DB (`NOTION_DEV_DB_ID = 30156ff9...`) 加：
  - `下次拜訪時間` (date) → TS: `nextVisitAt`
  - `行事曆ID` (rich_text) → TS: `calendarEventId`（實際存 Google Calendar htmlLink）
- 腳本：`scripts/_add-visit-fields.js`（dry-run + `--apply` 兩段式、跟既有 `_phase7-add-props.js` 同模板）
- `app/api/dev/route.ts` 同步：`DevPropertyData` interface + `mapPage` + `PATCH` 都加上對應欄位
- `components/PropertyDetailModal.tsx` 的 `DevProperty` 型別也補上、並 export `VisitTodo` 給 page 共用

### Ticket #2 — 委託 tab 內「開發」sub-tab 升級成 5 子分頁

- 把「開發信」sub-tab 改名為「**開發**」（不只信件、還有拜訪）
- sub-tab 內加 5 個子分頁、順序：`[拜訪物件地] [覆訪物件地] [拜訪戶藉地] [覆訪戶藉地] [待寄信]`
  - 預設停在「拜訪物件地」（日常高頻動作）
  - 「待寄信」內保留原本 `待寄 / 已寄留底` 切換 + 全選 + 產生 .docx 標籤完全不變
- 每個子分頁尾巴顯示計數（`devSubTabCounts` memo、僅算 `status='募集'`）
- 4 個拜訪 sub-tab 撈 `status='募集' AND visitTodo=<對應>`

### Ticket #3 — 拜訪卡片（純排程版、無 action 按鈕）

- 卡片內容：物件名 (連 Notion) / 屋主 / 手機 (可撥打) / 地址 (物件 or 戶藉、依 tab 自動切)
- 排程：用既有 `DateTimePopover` + 新加 `hour12` opt-in prop（12 小時制 + 上午/下午 dropdown 放在時間前面 + 15 分鐘顆粒）
- 「建立行事曆」按鈕 → `POST /api/dev/schedule-visit`
  - 後端串現有 `lib/mba/google-calendar.ts` 的 `createTimedEvent()`
  - event summary: `[拜訪物件地] 王大明 · 屋主：李小明`
  - event location 帶上地址、description 含手機 + Notion link
  - 回傳 `{ eventId, htmlLink }`、`htmlLink` 寫進 Notion「行事曆ID」欄
- 已排狀態：卡片顯示「已排 5/22 下午 3:00」+「開 Calendar」連結 + 改時間 picker
- `createTimedEvent` 簽名改成回 `{ id, htmlLink }`、`app/api/viewings/route.ts` 的 caller 一起修

### Ticket #4 — 戶藉地 tab 區域 filter chip

- 「拜訪戶藉地」「覆訪戶藉地」兩個 tab 內加 [高雄] [南部] [中北部] multi-select chip
- 預設只勾「高雄」(其他兩區點才展開、user 主要在高雄活動)
- 字串前綴判斷：
  - `高雄市/高雄縣/高雄` → 高雄
  - `嘉義/台南/臺南/屏東` → 南部
  - 其他（含空） → 中北部
- 每個 chip 尾巴顯示該區域物件數（過濾 `status='募集' AND visitTodo=<對應>` 後再分區）

### 取代 n8n

- 「待辦狀態變更 → 寫 Calendar」這條鏈被 CRM 直接覆蓋：CRM 卡片建立行事曆 → 直接呼 `createTimedEvent()` → 不過 n8n
- 對應的 n8n workflow（「Calendar grey 節點」前段）可以關掉

## 【為什麼這樣做】

### 北極星：一眼看到工作重點、不堆雜訊

user session 內連續糾正兩次：

1. 我在拜訪卡底部加灰字「完整拜訪卡片 — task #3 即將上線」當 placeholder、被打回（production UI 不放 dev/TODO 文字）
2. 我把 MBA 三按鈕全搬到 CRM 卡片、被打回說「兩個就好」、然後再被打回說「三個都不要」

**根本原因**：我一直把 CRM 拜訪卡當 pipeline 的一環、但 user 的設計是「CRM 拜訪卡 = 排程介面、不是現場操作介面」。職責分割很清楚、混了就違反北極星。

### 開發信 vs 拜訪 = 平行線

user 拍板「所有募集的頁面新增時、我都是預設拜訪物件地」——意思是 `visitTodo` 預設值由 **MBA 在新增募集物件時就設好**、不是 CRM 寄信流程觸發。所以 CRM 開發信頁面內、「待寄信」跟 4 個拜訪 sub-tab **不互相觸發**、各做各的。

### 為什麼用 chip 而不是 tab 做地區

地理是「客戶屬性」、不是「行為階段」、兩個維度應該分開。tab 數量爆掉沒意義、chip 可以多選、user 規劃南部一日掃路線時可以單獨展開「南部」chip 配合縣市排序。

### 為什麼 hour12 用 opt-in prop 而不是改默認

`DateTimePopover` 在 CRM 其他地方（行銷/委託）已經大量用 24h、user 對 24h 已經熟悉。為新拜訪卡加 prop = 不破壞既有用法 + 滿足新需求。

### 為什麼存 htmlLink 不存 eventId

Google Calendar event detail URL 需要 base64-encoded `{eventId} {calendarId}` 才能 deep-link、要在 client 端拼太脆弱。直接存 API 回傳的 `htmlLink`、user 點開即進該 event、最穩。Notion 欄名叫「行事曆ID」但實際存 URL、語意有點 off 但能 work、未來不爽再改名。

## 【下次接手要注意】

## 【為什麼這樣做】

- **MBA vs CRM 職責**：成功 path（找到→募集→追蹤）是 MBA 的事；CRM 拜訪卡純排程、零 action 按鈕（`[[project-crm-devletter-vs-visit-parallel]]`）
- **「X 跟 Y 一樣」= unify render**：本輪最大教訓，寫進 `[[feedback-x-like-y-means-unify]]`。看到「一致/一樣/同款」先停手確認「是不是兩 state 共用同一份 render」，不要急著 surgical 加減
- **成交客 雙身分不去重**：客戶同時買又賣、兩 DB 都標成交客 → 出現兩張同名卡。user 拍板「當沒看到」。否決新 DB+relation（= Phase 7 三層 schema 換名）與手機 join key 去重。理由：一年沒幾個 × 本來就認得 = 負 ROI。寫進 `[[feedback-solo-realtor-avoid-team-schema]]`
- **帶看寫回走 append log**：對齊 codebase 慣例（quick-log / blocks / notion-writeback 全是 append 帶時間戳活動 log），不另立單行同步（要加 schema 欄存 block id、不值得）
